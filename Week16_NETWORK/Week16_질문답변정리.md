# Week 16 - 질문/답변 정리

#### 권혁근
1. 흐름제어는 어떨 때 실행되고 혼잡제어는 어떨 때 실행되는가
```
흐름제어는 수신측이 송신측보다 데이터 처리 속도가 빠르면 문제없지만, 송신측의 속도가 빠를 경우 문제가 생긴다. 이때 데이터가 손실되지 않고 송수신이 다시 반복되지 않게 하기 위해 실행된다.
혼잡제어는 한 라우터에 데이터가 몰릴 경우, 자신에게 온 데이터를 모두 처리할 수 없게 된다. 이런 경우 호스트들은 또 다시 재전송을 하게되고 결국 혼잡만 가중시켜 오버플로우나 데이터 손신을 발생시키게 된다. 따라서 이러한 네트워크의 혼잡이 발생한 경우 실행된다.
```
2. Slow Start는 AIMD와 무엇이 다른가?
```
AIMD는 window size를 RTT마다 1씩 증가하지만 Slow Start는 2배씩 증가 시킨다. 패킷 손실이 일어나면 AIMD는 window size를 절반으로 감소시키지만 Slow Start는 1로 줄인다.
RTT(Round Trip Time) : 패킷을 목적지에 보낼 때, 패킷이 목적지에 도달하고 나서 해당 패킷에 대한 응답이 출발지로 다시 돌아오기까지의 시간 즉, RTT는 패킷 왕복 시간입니다.
```
3. UDP 통신이란?
```
User Datagram Protocol의 약자로 데이터를 데이터그램 단위로 처리하는 프로토콜이다.
비연결형, 신뢰성 없는 전송 프로토콜이다.
데이터그램 단위로 쪼개면서 전송을 해야하기 때문에 전송 계층이다.
```
4. DNS(Domain Name Service)에서 UDP를 사용하는 이유
```
TCP가 3-way handshake를 사용하는 반면, UDP는 connection 을 유지할 필요가 없음.
DNS request는 UDP segment에 꼭 들어갈 정도로 작음.
UDP는 not reliable이나, reliability는 application layer에 추가될 수 있음. (Timeout 추가나, resend 작업을 통해)
```
5. a가 b에게 전송할때 공개키 암호화 방식 진행 과정
```
A가 웹 상에 공개된 'B의 공개키'를 이용해 평문을 암호화하여 B에게 보냄
B는 자신의 비밀키로 복호화한 평문을 확인, A의 공개키로 응답을 암호화하여 A에개 보냄
A는 자신의 비밀키로 암호화된 응답문을 복호화함
```

#### 김빛누리
1. TCP, UDP 특징 / 사용하는 곳

```
TCP
연결형, 연결에서 3 way handshake / 4 way handshake 사용
흐름제어, 혼잡제어 사용
대부분의 HTTP 통신, 이메일이나 파일전송

UDP
비연결형, 신뢰성 X, 전송계층 프로토콜
데이터를 데이터그램 단위로 처리
실시간 시스템이나 온라인 게임 등 빠른 속도를 요구하는 곳에서 사용된다.

```

- DNS에서 UDP 쓰는 이유

```
TCP가 3-way handshake를 사용하는 반면, UDP는 connection 을 유지할 필요가 없음.

DNS request는 UDP segment에 꼭 들어갈 정도로 작음.

DNS query는 single UDP request와 server로부터의 single UDP reply로 구성되어 있음.

UDP는 not reliable이나, reliability는 application layer에 추가될 수 있음. (Timeout 추가나, resend 작업을 통해)
```

2. TCP에서 흐름제어, 혼잡제어 가 무엇인지

```
흐름제어
송신측과 수신측 사이의 데이터 처리 속도 차이를 해결하기 위한 기법이다
송신 측의 전송량 > 수신측의 처리량인 경우, 패킷이 수신 측의 큐를 넘어 손실될 수 있어서 
송신측 패킷 전송량을 제어해줘야 한다.

혼잡제어
네트워크 혼잡을 피하기 위해 송신 측에서 보내는 데이터의 전송 속도를 제어하는 것
송신 측 데이터 전달과 네트워크 데이터 처리 속도 차이를 해결하기 위한 기법
한 라우터에 데이터가 몰려서 모든 데이터를 처리할 수 없는 상황에 이르렀을 때, 호스트들은 계속해서 재전송을 하고 결과적으로 혼잡만 가중시켜 오버플로우나 데이터 손실이 발생한다.
```

3. 흐름제어, 혼잡제어 방식 아는대로

```
흐름제어 : Stop and Wait, Sliding Window
혼잡제어 : AIMD, Slow Start, 빠른 재전송, 빠른 회복
```

- 혼잡제어 방식 하나 설명

```
Slow Start (느린 시작)
- AIMD 방식이 네트워크의 수용량 주변에서는 효율적으로 작동하지만, 
처음에 전송 속도를 올리는데 시간이 오래 걸리는 단점

- 처음에는 AIMD와 같지만 한번 혼잡 현상이 발생하고 나면 네트워크의 수용량을 어느 정도 예상

- 전송속도는 AIMD에 반해 지수 함수 꼴로 증가한다. 대신에 혼잡 현상이 발생하면 
window size를 1로 떨어뜨리게 된다. 그 이후부터는 완만하게 1씩 증가시킨다.
```

4. 대칭키, 공개키 + 예시

```
대칭키(Symmetric Key): 암호화와 복호화에 같은 암호키(대칭키)를 사용하는 알고리즘, 속도 빠름
공개키(Public Key): 암호화와 복호화에 사용하는 암호키를 분리한 알고리즘, 속도 느림, 보안
```

#### 박상민
1. TCP/IP 흐름제어 기법에는 무엇이 있나요?
```
Stop and Wait
전송한 패킷에 대해 확인 응답을 받으면 다음 패킷을 전송하는 제어 기법
Sliding Window
수신측에서 설정한 윈도우 크기만큼 송신측에서 확인응답 없이 패킷을 전송할 수 있게 하여 데이터 흐름을 동적으로 조절하는 제어 기법
```
2. TCP/IP 혼잡 제어 기법은 왜 사용이 되나요?
```
송신측에서 보내는 데이터 양을 라우터가 감당하지 못하여 누락시키면 송신측에서는 손실 데이터로 간주하여 계속 재전송을 하여 네트워크를 혼잡하게 한다.
이러한 상황을 예방하기 위해서 송신측의 전송 속도를 적절히 조절하는 혼잡 제어 기법이 사용된다.
대표적으로 AIMD(선형적증가), Slow Start (지수적증가), 빠른 재전송, 빠른 회복 등이 있다.
```
3. UDP 의 특징에는 무엇이 있나요?
```
비연결성이고 신뢰성 없는 전송계층의 프로토콜
데이터처리가 TCP 보다 빠르다.
실시간 방송과 온라인 게임에서 사용된다.
```
4. UDP 를 사용하는 어플리케이션 계층의 프로토콜 중에는 DNS 가 있는데 DNS 는 왜 UDP 를 사용하나요?
```
DNS 의 request 가 UDP segment 에 들어갈 만큼 작기 때문에 사용
하지만 DNS 서버 간 요청을 주고 받는 Zone transfer 의 경우 또는 데이터의 크기가 512 bytes 를 넘기는 경우에는 TCP 를 사용해야한다.
```
5. 대칭키와 공개키 방식의 차이점은 무엇인가요?
```
대칭키는 암호화와 복호화에 같은 암호키를 사용하는 알고리즘이고, 공개키 방식은 암호화와 복호화에 사용하는 암호키를 분리한 알고리즘입니다. 암호화는 공개키를 사용하여 하고 복호화는 비밀키로만 가능합니다.
```
6. 대칭키 암호화 방식 단점은 무엇인가요?
```
중간에 키를 탈취당했을 때 위험에 노출된다는 단점이 있습니다.
```
7. 그럼 그 단점을 보완하기 위한 해결방법이 있나요?
```
대칭키를 공개키 알고리즘으로 암호화하여 전달함으로써 키 분배문제를 해결할 수 있습니다.
```
8. Https 란 무엇인가요?
```
기존의 Http 를 ssl 프로토콜을 사용해 암호화하여 서버와 클라이언트간 자원을 주고 받도록하는 통신규약
```

#### 장시우
1. TCP와 UDP란 무엇이며 차이점은 무엇인가?
```
TCP : 네트워크 통신에서 신뢰적인 연결방식
TCP는 기본적으로 unreliable network에서, reliable network를
보장할 수 있도록 하는 프로토콜
TCP는 network congestion avoidance algorithm을 사용
UDP : 비연결형, 신뢰성이 없는 전송 프로토콜
UDP는 IP데이터그램을 캡슐화하여 보내는 방법과 연결 설정을
하지 않고 보내는 방법을 제공한다.
UDP는 TCP/IP 5계층에서 Transport Layer(전송계층)의 프로토콜
```
2. 대칭키와 공개키란 무엇인가?
```
대칭키 : 암호화와 복호화에 같은 암호키(대칭키)를 사용하는
알고리즘, 동일한 키를 주고받기 때문에, 매우 빠르다는 장점이 있음.
하지만, 대칭키 전달과정에서 해킹 위험에 노출
공개키/비대칭키 : 암호화와 복호화에 사용하는 암호키를 분리한
알고리즘, 대칭키의 키 분배 문제를 해결하기 위해 고안됨.
자신이 가지고 있는 고유한 암호키(비밀키)로만 복호화할 수 있는
암호키(공개키)를 대중에 공개함
```
3. HTTPS란 무엇인가?
```
인터넷 상에서 정보를 암호화하는 SSL 프로토콜을 사용해 클라이언트와 
서버가 자원을 주고 받을 때 쓰는 통신 규약
```
